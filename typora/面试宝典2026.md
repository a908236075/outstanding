# 面试宝典

## JVM

### 逃逸分析

逃逸分析并不是一种垃圾回收（GC）算法，而是 JIT（即时编译器）在编译期间执行的一种代码优化技术

#### 1. 逃逸分析的三种状态（对象去了哪里？）

JIT 编译器在分析代码时，会将对象的作用域分为三个级别：

- **不逃逸 (No Escape)：** 一个对象在方法内部被创建，且它的所有引用永远都不会被传递到该方法之外，也没有被其他线程访问到。这是**最完美的优化目标**。
- **方法逃逸 (Method Escape)：** 一个对象在方法内部被创建，但它作为返回值传给了调用者，或者作为参数传递给了其他方法。
- **线程逃逸 (Thread Escape)：** 一个对象不仅逃出了方法，甚至逃出了当前线程。比如把它赋值给了一个 `static` 静态变量，或者放入了多个线程共享的集合中。

## 零拷贝

“零拷贝”并不代表完全不拷贝数据，而是指“不需要 CPU 参与”的数据拷贝（即没有 CPU 拷贝），并且尽量减少用户态与内核态的上下文切换。

## 线程

### 高并发的理解以及怎么提高并发

1. **高并发（High Concurrency）** 是指系统在同一时间段内能够同时处理大量的请求

2. 提高并发本质上是“空间换时间”或“异步解耦”的艺术。我们可以从以下四个核心维度进行优化：

   #### 1. 扩容：提升物理上限

   - **垂直扩容（Vertical Scalability）**：提升单机配置（更强的 CPU、更大的内存、NVMe SSD）。
   - **水平扩容（Horizontal Scalability）**：通过集群化部署，配合**负载均衡（Nginx / LVS / F5）**将请求分发到多台服务器。

   #### 2. 缓存：减少 IO 损耗

   缓存是解决高并发的“银弹”，它将数据从磁盘（慢）移动到内存（快）。

   - **应用级缓存**：Redis、Memcached。
   - **本地缓存**：Caffeine、Guava Cache。
   - **浏览器/边缘缓存**：CDN 加速，将静态资源推送到离用户最近的节点。

   #### 3. 异步：削峰填谷

   将同步处理流程改为异步，避免主线程被耗时任务阻塞。

   - **引入消息队列（MQ）**：如 Kafka、RocketMQ。当流量大时，先将请求写入 MQ，后端慢慢消费。
   - **响应式编程**：利用 CompletableFuture 或 WebFlux 提高单机的线程利用率。

   #### 4. 存储层优化：解决最终瓶颈

   数据库往往是高并发系统的最后一道防线。

   - **读写分离**：主库负责写，从库负责读。
   - **分库分表**：解决单表数据过大导致的索引性能下降。
   - **SQL 优化**：确保索引命中，避免全表扫描。
   - **连接池优化**：合理配置 Druid 或 HikariCP 的参数，防止连接耗尽。

3. 代码层面:

   1. 使用多线程
   2. 接口设置为无状态的.

### 多节点本地缓存同步方案

由于本地缓存存在于每个应用的 JVM 内存中，当某个节点修改了数据，其他节点的缓存仍是旧值。以下是几种常见的同步/更新策略：

#### 1. 订阅发布模式（推荐：MQ 方案）

这是最常用且解耦最好的方案。

- **原理**：当某个节点修改了数据库中的数据后，向消息队列（如 Kafka、RocketMQ、RabbitMQ）发送一个广播消息（Topic）。
- **执行**：所有订阅了该 Topic 的节点（包括修改者自己）收到消息后，各自失效（Invalidate）或更新本地缓存。
- **优点**：实时性较好，支持水平扩展。

#### 2. 结合分布式缓存（二级缓存方案）

构建 **本地缓存 (L1) + Redis (L2)** 的两级架构。

- **查询路径**：先查本地缓存 $\rightarrow$ 无则查 Redis $\rightarrow$ 无则查数据库。
- **同步策略**：
  - **主动过期**：设置本地缓存的 TTL（过期时间）非常短（如 1 分钟）。
  - **版本号比对**：在 Redis 中存放数据的一个版本号或最后更新时间。节点先对比本地版本号和 Redis 版本号，若不一致则更新本地缓存。

#### 3. 注册中心监听（ZooKeeper / Nacos）

利用注册中心或配置中心的监听（Watch）机制。

- **执行**：将配置或数据元信息存放在 Nacos/ZK 上。当数据变化时，配置中心会通过长轮询或推送机制通知所有应用实例，触发本地缓存的刷新。
- **优点**：对于“配置类”数据的同步非常成熟。

#### 4. 固定时间过期（最简单的方案）

- **原理**：不进行显式同步，而是为每个节点的本地缓存设置较短的过期时间（如 `expireAfterWrite(30, TimeUnit.SECONDS)`）。
- **缺点**：在过期前的这段时间内，各节点间会出现数据不一致，仅适用于对实时性要求较低的场景。

## 事务

###  实现最终一致性的四大常见模式

在工业界，实现最终一致性没有银弹，通常根据业务场景的对“一致性延迟的容忍度”和“开发成本”来选择以下四种模式：

#### 1. TCC (Try - Confirm - Cancel) 模式

TCC 属于**业务层面的两阶段提交**。它要求业务代码把一个操作拆分成三个独立的方法。

- **Try：** 资源的检测和预留（锁定）。比如：冻结用户账户里 100 块钱，而不是直接扣除。
- **Confirm：** 真正的业务提交。如果所有服务的 Try 都成功了，执行 Confirm。比如：把冻结的 100 块钱真实扣掉。
- **Cancel：** 业务回滚。如果任何一个服务的 Try 失败了，执行 Cancel。比如：解冻那 100 块钱。
- **核心特点：** 强依赖业务代码，侵入性极高，但一致性保障非常好，几乎没有长期的中间状态。

#### 2. Saga 模式

Saga 模式把一个全局分布式事务，拆解成一长串的**本地事务（$T_1, T_2, T_3 \dots$）**。

- 每个本地事务 $T_i$ 都有一个对应的**补偿动作 $C_i$**。
- **正向执行：** 依次执行 $T_1 \rightarrow T_2 \rightarrow T_3$。
- **反向回滚（如果 $T_3$ 报错了）：** 系统会自动按相反顺序调用补偿逻辑，即执行 $C_2 \rightarrow C_1$，把之前做过的事情“撤销”掉。
- **核心特点：** 适合耗时极长（比如需要跨天、跨机构交互）的事务。它没有 Try 的资源锁定阶段，所以可能会发生“脏读”（别人看到了你做到一半的数据）。

#### 3. 本地消息表 (Local Message Table)

这是 eBay 早期提出的一种极其经典的可靠消息模式。

- **核心逻辑：** 业务侧在执行业务数据写入的同时，在**同一个数据库**里再写一张“消息表”。因为在同一个库，所以利用本地数据库事务保证了“业务成功”和“消息落库”同时发生。
- **后续动作：** 后台起一个定时任务（定时器），不断扫描这张消息表，把消息发给 MQ（比如 Kafka/RabbitMQ），让下游去消费。下游消费成功后，回调把消息表里的状态改为“已完成”。
- **核心特点：** 极度通用，不挑中间件，但会让业务库混入消息表，且定时轮询对数据库有性能压力。

#### 4. 事务消息 (Transactional Message)

这是本地消息表的“中间件升级版”，最典型的代表是 **RocketMQ** 的事务消息。

- **核心逻辑：**
  1. 业务先发一条“半消息（Half Message）”给 MQ（此时下游看不见）。
  2. MQ 收到后，回调业务执行本地数据库事务。
  3. 如果本地事务成功，发 Commit 给 MQ（下游可见）；如果失败，发 Rollback（抛弃消息）。
  4. **兜底回查：** 如果发送 Commit/Rollback 阶段网络断了，MQ 会主动来问业务：“你刚才那个本地事务到底成没成啊？”
- **核心特点：** 极其优雅，把定时轮询的压力交给了 MQ 中间件，解耦彻底。

## 数据库

###  深度分页的 4 种主流解决方案

面对深度分页，业界通常有以下几种处理方案，具体取决于你的业务场景是否允许妥协：

#### 方案 1：业务层面的妥协（限制最大分页数）—— 最推荐

很多时候，技术上的难题可以通过产品交互来化解。

- **做法：** 直接限制用户能够翻页的最大深度。
- **现实案例：** * 百度/谷歌搜索，通常最多只能往后翻 10 到 20 页。
  - 淘宝/京东的商品搜索，最多只能滑到第 100 页（比如最多展示 10000 个商品）。
- **逻辑：** 一个正常的用户如果翻了 100 页还没找到想要的东西，说明他的搜索条件有问题，你应该引导他增加筛选条件，而不是让他无限往下翻。

#### 方案 2：延迟关联（子查询/覆盖索引优化）

如果你必须支持跳页（比如直接跳到第 50000 页），可以通过“覆盖索引”来大幅提升性能。

- **传统慢 SQL：** `SELECT * FROM order_table WHERE status = 1 ORDER BY create_time LIMIT 100000, 10;` (这会发生 100000 次回表，极慢)

- **优化后 SQL：**

  SQL

  ```
  SELECT t1.* FROM order_table t1
  INNER JOIN (
      -- 内部这个子查询因为只查主键 id，会走覆盖索引，完全在内存中完成，速度极快
      SELECT id FROM order_table WHERE status = 1 ORDER BY create_time LIMIT 100000, 10
  ) t2 ON t1.id = t2.id;
  ```

- **原理：** 先通过极快的内部子查询（利用覆盖索引，不回表）定位到那 10 条目标数据的 ID，然后拿着这 10 个 ID 去主表里取完整数据。

#### 方案 3：游标分页法（记录上一页的最大/最小值）

如果你的产品交互不需要“跳页”（即没有 1, 2, 3, 4, 5 的页码），而是类似于朋友圈的**“下拉加载更多”**，游标法是性能最完美的方案。

- **做法：** 每次查询时，带上上一页最后一条记录的标志位（通常是自增 ID 或时间戳）。

- **优化后 SQL：**

  SQL

  ```
  -- 假设前端传来了上一页最后一条记录的 ID 是 1000000
  SELECT * FROM order_table WHERE id > 1000000 ORDER BY id ASC LIMIT 10;
  ```

- **原理：** 数据库直接利用 B+ 树索引，精准定位到 `id = 1000000` 的位置，然后往后扫 10 条。完全没有 `offset` 带来的扫描废弃数据的问题，无论翻多深，查询时间永远是常量级别（O(1)）。

#### 方案 4：借助搜索引擎（Elasticsearch）

对于极其复杂的全文检索或多维度筛选，关系型数据库本来就不擅长。

- **做法：** 将数据同步到 Elasticsearch。
- **ES 的解决方案：** * 虽然 ES 默认的 `from + size` 也有深度分页上限（默认 10000）。
  - 但 ES 提供了 `search_after` API（基于游标机制，类似方案3）或者 `Scroll` API（用于导出海量数据），专门解决深分页问题。

### 一、 死锁是怎么出现的？（两大经典场景）

死锁的本质是**“循环等待”**：两个或多个事务各自占有一部分锁，同时又在互相请求对方手里的锁，谁也不肯放手，最终僵持住。

#### 1. 最常见的元凶：更新顺序不一致 (交叉交叉锁)

这是 80% 业务代码产生死锁的原因。

- **事务 A**：先更新订单 1，再更新订单 2。
- **事务 B**：先更新订单 2，再更新订单 1。
- **发作过程**：
  1. 事务 A 锁住了订单 1 的行。
  2. 与此同时，事务 B 锁住了订单 2 的行。
  3. 事务 A 想要去锁订单 2，发现被 B 占了，于是等待。
  4. 事务 B 想要去锁订单 1，发现被 A 占了，于是等待。
  5. **💥 砰！死锁形成。**

#### 2. 隐蔽的杀手：间隙锁 (Gap Lock) 冲突

在默认的 `REPEATABLE READ`（可重复读）隔离级别下，为了防止幻读，InnoDB 会使用间隙锁。

- 假设表里有 id 为 10 和 20 的记录。
- **事务 A** 执行 `DELETE FROM table WHERE id = 15;`。虽然 id=15 不存在，但 A 会锁住 (10, 20) 这个**间隙**。
- **事务 B** 也执行同样的操作，或者尝试在这个间隙内 `INSERT` 一条数据。
- 多个事务可以同时获得同一个间隙的间隙锁，但当它们随后尝试往这个间隙里插入数据时（请求插入意向锁），就会互相阻塞，导致死锁。

### 二、 遇到死锁怎么排查和解决？

既然知道了原理，解决思路也就清晰了。排查和解决通常分为三步走：

#### 第一步：保留现场（定位问题代码）

不要靠猜，直接让 MySQL 告诉你发生了什么。 登录 MySQL，执行这行命令：

SQL

```
SHOW ENGINE INNODB STATUS;
```

找到 `LATEST DETECTED DEADLOCK`（最近检测到的死锁）这一段日志。里面会非常详细地记录：

- **事务 1** 正在执行什么 SQL，持有哪把锁，在等哪把锁。
- **事务 2** 正在执行什么 SQL，持有哪把锁，在等哪把锁。 拿着这些 SQL 语句，回溯到你的 Java/Go 业务代码里，就能精准找到是哪两个方法在打架。

#### 第二步：根治死锁（架构与代码优化）

针对上面提到的场景，对症下药：

1. **固定访问顺序（最有效）：** 如果你的业务需要同时更新多个资源，**强制要求所有代码按相同的顺序去访问**。比如，总是先按订单 ID 从小到大排序，然后再依次执行 `UPDATE`。只要顺序一致，即使并发再高，也只会发生排队等待，绝不会死锁。
2. **大事务化小（减少锁定时间）：** 事务越长，持有的锁就越多，死锁的概率呈指数级上升。把非核心的查询移出 `@Transactional` 事务块，只把最核心的 `UPDATE/INSERT` 放在事务里。
3. **精细化索引，避免全表锁：** 如果你的 `UPDATE` 语句的 `WHERE` 条件没有走索引，MySQL 会把整张表的所有行都锁住！确保每一条更新语句都精准命中索引（最好是主键或唯一索引）。
4. **降低隔离级别（杀手锏）：** 如果是刚才提到的“间隙锁”导致的死锁，且你的业务并不怕“幻读”（比如大多数电商订单流转场景），可以将 MySQL 的事务隔离级别从默认的 `REPEATABLE-READ` 降级为 `READ-COMMITTED`。这样会极大减少间隙锁的使用，直接从底层抹除一大批死锁隐患。

#### 第三步：业务层面的兜底防御

在分布式高并发系统中，完全杜绝死锁是不现实的。我们必须在代码里做好异常捕获： 当捕获到 `Deadlock found when trying to get lock` 异常时，不要直接给前端抛错。可以在代码里稍微 `sleep` 几十毫秒，然后**重试 1-3 次**。因为 MySQL 已经解开了死锁，重试大概率会成功。
